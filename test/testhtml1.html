DESCRIPTION

A nextrieve resource file is a simple text file containing information in XML form about or related to a text index.

A DTD describing the resource file format can be found here.

The structure of a resource file is simple; there is an outer <ntv:resource> container which contains a small number of other top-level containers:

miscellaneous
There are a small number of miscellaneous elements specifying information that's generally useful to all nextrieve products regarding this text index.

indexcreation
The <indexcreation> container collects all the information related to the structure of a text index. If any information in this section is changed, the text index should be rebuilt with ntvindex from original text sources.

indexing
The <indexing> container holds the information used by ntvindex as it performs indexing operations. This is information relating to what to do with various error conditions, etc.

searching
The <searching> container holds the information used by ntvsearch and ntvsearchd to determine how to perform searches and how to return results.

ultralite
The <ultralite> container(s) are solely used by ntvultralite, nextrieve's HTML template filling process.

All the containers are EMPTY containers, ie, no text is expected in a container.

In summary, the top-level structure of a nextrieve resource file looks something like this:

 
 
    <ntv:resource xmlns:ntv="http://www.nextrieve.com/1.0">


        <misc1 .../>
        <misc2 .../>
        ...


        <indexcreation>
            <index-creation-specific.../>
            ...
        </indexcreation>


        <indexing>
            <ntvindex-specific.../>
            ...
        </indexing>


        <searching>
            <nextrieve-specific.../>
            ...
        </searching>


        <ultralite>
            <ntvultralite-specific.../>
            ...
        </ultralite>


    </ntv:resource>

The content of the resource file is discussed according to these classifications. Within each classification, containers are presented in no particular order.

Miscellaneous Elements

<licensefile name='licensefile'/>
<licencefile name='licensefile'/>
These elements are synonyms of each other. If one is specified, it indicates the full name of the nextrieve licensefile on the system. If neither of these lines are present in the resource file, a set of standard locations is searched, namely /usr/local/nextrieve/licen[cs]e and /etc/nextrieve/licen[cs]e.

<indexdir name='index-directory'/>
index-directory is the name of the directory in which all the text-index-related files are found. These files all end with the suffix ``.ntv''.

<logfile name='logfile-name'/>
During its operation, the indexer (ntvindex) or searcher (ntvsearch or ntvsearchd) may print various time-stamped messages. These messages are printed to stderr (the standard error output) and, if specified, to the file specified as logfile-name.

<cache size=``cachesize''/>
A memory cache is used during indexing and during searching, to increase performance. This line specifies the amount of memory to allocate for the cache. cachesize is a number which can be suffixed with K, M or G indicating kilobytes, megabytes or gigabytes. If only a number is used for cachesize, it is taken to be the size in bytes.

By default a value of 10M (10Mb) is used, but normally a much larger value is specified, dependant somewhat on how large the index is and how much physical memory is available on the indexing machine. Values of 100M are not uncommon.

<indexcreation> Container

The containers permitted in an <indexcreation> container are as follows:

<exact [accentaction=``distinct''|``merge''|``both'']/>
<fuzzy [accentaction=``distinct''|``merge''|``both'']/>
The <exact> element indicates that the text index will contain an ``exact'' part. The <fuzzy> element indicates that the text index will contain a fuzzy part.

With an exact index it is possible to search for words in an exact fashion. Only words that exactly match words in a particular search query are relevant.

With a fuzzy index it is possible to search for words in a fuzzy fashion. Words that are similar to original query words can be considered relevant. A ``fuzzy'' level, specified in the search, gives an indication of the permitted word variation.

It is possible to have both exact and fuzzy in the same index. In fact, if both are present, the accuracy and quality of fuzzy searches is significantly improved by the information found in the exact index.

The optional accentaction attribute specifies what to do with accented characters, as defined by the <utf8data foldfilename=...> tag.

distinct
Accented characters are ``preserved''.

For example, if the word ``voilà'' is indexed, it can only be found in an exact index by searching for ``voilà'' (with the accent). In a fuzzy index, searching for ``voilà'' will consider ``voilà'' to be an exact match, and ``voila'' (without the accent) to be ``further away''.

merge
Accented characters are ``decomposed'' to their closest non-accented equivalents.

For example, if the word ``voilà'' is indexed, it will be stored in the index as ``voila'' (no accents). It will be found by an exact or fuzzy search for either ``voilà'' or ``voila'', which both map automatically to a search for ``voila'' (no accents).

both
Both the above operations are performed.

If the word ``voilà'' is indexed, it is stored in the index as both ``voilà'' and ``voila''. During searching, if the word ``voila'' (no accent) is searched for, both ``voila'' and ``voilà'' will therefore be found. If the word ``voilà'' (accented) is searched for in an exact fashion, only occurrences of ``voilà'' will be found. A fuzzy search for ``voilà'' will treat ``voila'' to be ``further away''.

In this way a search using non-accented text will find both non-accented and accented forms of the words. A search using explicitly accented text will find only the accented forms that were indexed.

It should be noted, however, that in the text preview provided as part of the search results, accented characters are displayed in their original form. The lines discussed above relate only the searching of the text as a consequence of how it was indexed.

By default, an accentaction of ``both'' is used for both exact and fuzzy indexes. It is recommended that if both an exact and fuzzy text index are being created, that the accentaction of both is the same.

<texttype name=``ttname''/>
This element specifies the name of an allowable ``text type''.

The <text> element of a document being indexed can contain nested containers that are indexed specially. If the name of a nested container matches one of the allowable texttype names (as defined with a <texttype> element), the text contained in that container will be indexed in a manner that makes it searchable separately. Eg,

    <texttype name="heading"/>

allows the use of the following in the text of a document being indexed:

 
 
    <text>
        normal text.
        <heading>title text</heading>
        more normal text.
        <heading>more heading text</heading>
        more normal text.
    </text>

It is then possible to search the text index looking at only the heading text, for example. It is also possible to search the text index looking at all text, but giving text found in <heading> containers a higher weight. By default, all text types are searched.

A discussion of search operations can be found here.

The number of different text types is somewhat limited -- they should be chosen with reasonable care.

<attribute name=``attrname'' type=``flag''|``number''|``string'' key=``key-unique''|``key-duplicates''|``notkey'' nvals=``1''|``*''/>
An <attribute> element specifies the characteristics of a particular per-document attribute.

The name of the attribute is given by attrname.

Attributes for a document are presented to the indexer in the XML input stream and are processed at the same time as the document text is indexed. Each attribute has a name by which it is referenced, and a value. The value can be one of a number of different types:

type=``flag''
The value of the attribute can be either 0 (off) or non-zero (on). If the value of a particular flag attribute is not specified for a document, it is taken to be ``off''. Similarly, if the value of a flag attribute is explicitly specified as: 
 
 
    <attrname>0</attrname>

the value is taken to be ``off''. If the value is specified as one of:

 
 
    <attrname/>
    <attrname></attrname>
    <attrname>1</attrname>

the value is taken to be ``on''.

type=``number''
The value of the attribute can be a non-negative number in the range 0..2147483647 inclusive.

If the value of a number attribute is not specified for a document, the value is taken to be zero.

type=``string''
The value of the attribute can be any string of characters.

It should be noted that the values of these attributes cannot be searched in the way document text can be. The values of these attributes can only be:

  • Be used as part of a constraint, restricting the set of valid documents to return for a search result.

  • Returned as part of search results, giving information about the document found.

The ``key'' parameter indicates the following characteristics of a document attribute:

key=``key-unique''
A ``key-unique'' parameter implies that only one document in the text index can have any particular attribute value. If, during indexing, a second document is presented with an attribute value of attrname that is already associated with a document in the index, the existing document is automatically marked deleted.

key=``key-duplicates''
A ``key-duplicates'' attribute will eventually allow fast retrieval of documents when only the value of attrname is specified. In the current release of NexTrieve ``key-duplicates'' is accepted but is normally only used for string attributes where there are many occurrences of the same per-document value: a side-effect of using ``key-duplicates'' is that values are ``shared'' amongst documents. All documents having the same string value all point to the same piece of text, reducing memory requirements if the strings are generally long and duplicated.

key=``notkey''
Attribute values are not keyed or shared in any way.

The ``nvals'' parameter indicates if a single document can have only at most one value of the attribute, or many:

nvals=``1''
A particular document can only have one value of this particular attribute. A ``filename'' attribute is an example.

nvals=``*''
A particular document can have several values associated with this particular attribute. A ``keyword'' or ``author'' attribute is a good example. In the XML stream being indexed, the multiple values of such an attribute are defined simply by repeating the <attrname>...</attrname> specification, eg: 
 
 
    <document>
        <attributes>
            <attrname>value1</attrname>
            <attrname>value2</attrname>
            <attrname>value3</attrname>
        </attributes>
    ...

These per-document attributes are reported as part of the search results for a query. They can also be used as part of a ``constraint'' in a search, an expression built up out of attribute names and values that restrict the set of allowable documents returned as the result of a query. A discussion of constraints in nextrieve can be found here.

For a flag-type attribute, the only valid settings for the other parameters are 'key=``notkey'' nvals=``1'''. Other variations for a flag type attribute will return an error.

<utf8data classfile=``classfilename'' foldfile=``foldfilename'' decompfile=``decompfilename''/>
This element specifies names of files, files whose content overrides compiled-in content of nextrieve utilities. The functions are all related to unicode encoding. The compiled-in content of the nextrieve utilities has been extracted from publicly available standards datafiles CaseFolding.txt and UnicodeData.txt.

For the file format of these files, refer to the section ``UTF8 Data Files'' below.

Indexing Container

The elements permitted in an <indexing> container are as follows:

<unknowntext logaction=``log''|``!log''|``stop'' indexaction=``ignore''|``default''/>
<nestedtext logaction=``log''|``!log''|``stop'' indexaction=``ignore''|``inherit''/>
During indexing of the <text> container of a document, it is possible for the ntvindex to find containers that are unknown or too deeply nested. unknowntext and nestedtext specify what to do in those situations.
logaction=``log''
A message is logged and indexing continues.

logaction=``!log''
A message is not logged and indexing continues.

logaction=``stop''
A message is logged and indexing stops.

indexaction=``ignore''
The text contained in the offending container is simply ignored.

indexaction=``default''
The text contained in the unknown container is indexed as if it was part of the top level default text of the document.

indexaction=``inherit''
If a text element is nested too deeply, the text it contains is indexed as if it was part of its immediate parent.

<unknownattrs logaction=``log''|``!log''|``stop''/>
<nestedattrs logaction=``log''|``!log''|``stop''/>
Similar to the <unknowntext> and <nestedtext> elements described above, these containers specify what to do with unknown or nested elements in the <attributes> section during indexing.

The offending elements are never indexed, but they can trigger messages and/or termination of indexing.

Searching Container

The elements permitted in a <searching> container are as follows:

<<highlight name=``tagname'' on=``ontext'' off=``offtext''>/>
If preview text is being returned in a hit list, words present in the preview text relevant to the original search terms in the preview are highlighted. By default, nextrieve will surround such terms with a <b></b> pair. It is possible to supply alternate highlighting text using the <highlight> element. If name=``tagname'' is given, nextrieve will surround words to be hightlighted with 
 
 
    <tagname>word</tagname>

Alternatively, if 'on=``ontext''' and 'off=``offtext''' are given, the text ``ontext'' and ``offtext'' will be used directly to introduce and leave the highlighted word respectively.

<hitlistxml form=``long''|``short''/>
When searching, the XML for the hitlist can be returned in a ``long'' form (<hitlist>...</hitlist>) or a short form (<hl>...</hl>). The <hitlistxml> element specifies the default if the form wanted is not specified in the query.

If the <hitlistxml> element is not present, a default of form=``long'' is applied.

<threads connector=``nconn'' worker=``nworker'' core=``ncore''/>
The <threads> element specifies the number of threads and connections allowed in a multi-threading ntvsearchd.

nconn defines the number of simultaneous client connections allowed to ntvsearchd. If the system supports threads and the server has been compiled with threading enabled, the request data arriving for each connection is read by a dedicated thread, otherwise the request data is read by a central loop. This can be a number around 50, for example.

nworker is only relevant for an ntvsearchd compiled multi-threaded. It defines the maximum number of simultanous requests able to be active in the server at any one time. This can have a value similar to nconn.

ncore is only relevant for an ntvsearchd compiled multi-threaded. It defines the maximum number of simultanous requests able to be in the inner decode loop at any one time. This is typically a small value such as 5.

To determine if ntvsearchd has been compiled multi-threaded, use the -V option. The number of threads and connections allowed is also printed to the logfile by ntvsearchd as the first log message after startup.

<querylog path=``qlogpath''/>
qlogpath specifies a directory in which ntvsearchd will create a file to be used for logging all search requests. The name of the file is constructed automatically from the current date and time when ntvsearchd starts up.

Ultralite Containers

The <ultralite> container is used by ntvultralite, nextrieve's HTML template filling application.

Different instances of ntvultralite are given different values of the environment variable $NTVNAME. The value of this variable is used for, among other things, to locate an <ntvultralite> element in the resource file with an attribute of name=``$NTVNAME''. If this container does not exist, ntvultralite will read the ``default'' container specified simply as <ntvultralite> with no ``name'' attribute.

The elements permitted in an <ultralite> container are as follows:

<attrmap name=``attrname'' text=``text''/>
Used by the <ntv-attrmap*> tag in a template. Arbitrary HTML, specified here as text, is substituted into the template if the nominated attribute attrname is set on a particular hit.

<fuzzybutton level=``n'' text=``text''/>
Used by the <ntv-ctl-fuzzyfactor> tag when it is filled with a <select> HTML element. The fuzzy level n is given select text of text.

<server name=``host:port''/>
If this container is present, ntvultralite will communicate with the nextrieve search server nominated as host and port to perform the query.

If this container is not present, ntvultralite will perform the query directly on the text index.

<vblsub name=``vblname'' text=``text''/>
<vblrange name1=``vblname1'' name2=``vblname2''/>
<vbluse name=``vblname'' type=``constraint|''text`` class=''any``|''all|``not|''free``|''ntv-vblname2``/>
A discussion of the use of ``variables'' in ntvultralite can be found here.

<hitlimit total=``maxth'' displayed=``displayed''/>
The <hitlimit> element restricts the maximum number of total hits and displayed hits asked for by a form to the values specified as maxth and displayed respectively.

By default total=``1000'' and displayed=``100'' are applied.

<emitok value=``0''|``1''/>
If present with a value of ``1'', ntvultralite will emit an ``HTTP/1.0 200 OK'' line in the HTTP header.

<log value=``0''|``1''/>
If present with a value of ``1'', ntvultralite will automatically create or append to a logfile named $NTVBASE/$NTVNAME/logs/ultralite.txt.

<execallow value=``0''|``1''/>
If present with a value of ``1'', the use of <ntv-exec> tags is permitted in templates when ntvultralite is filling them.

<switchedtemplate keyword=``kwd'' tempatedir=``templatedir''/>
A simple keyword scan is performed on the query string -- if the query string contains the word kwd, templates will be taken from the subdirectory templatedir under $NTVBASE/templates.

Note that switchedtemplate is only honored if there are no <attributetemplate> containers in the resource file.

<selectname name=``attrname'' text=``displayname''/>
This maps a flag-type attribute named attrname to text that will be displayed in a select box on an HTML form. Only required if <ntv-select> tags are used in templates.

<attributetemplate attrname=``attrname'' templatedir=``templatedir''/>
A particular subdirectory under $NTVBASE/templates/ (named by templatedir) can be nominated to be used if a particular flag-type attribute (named by attrname) has been set on the search form. This is intended for use when there is a single attribute set per form, set by using <ntv-select> for example. If multiple attributes are present, the first attributetemplate element having a match on the form is used. The template files within this subdirectory follow the normal ntvultralite file naming naming convention for templates.

<logicalindex name=``idxname''/>
If present, ntvultralite will fill in an <indexname> container in the query specifying that this logical text index is to be searched. This is useful where there are multiple ``logical'' text indexes in a single ``physical'' text index, managed by a nextrieve caching server.

UTF8 Data Files

As part of a <utf8data> element in the resource file:

 
 
    <utf8data classfile="classfilename"
              foldfile="foldfilename"
              decompfile="decompfilename"/> 



up to three files can be specified whose content will replace compiled-in
content of nextrieve utilities.

classfilename

A classfilename specifies a file whose content defines the classification of unicode data. The first line of this file gives the value (in hex) of the maximum unicode code point of an acceptable character. Subsequent lines give the unicode code point (in hex) of a character, followed by a space, followed by it's nextrieve-specific classification. This classification is a decimal number representing the sum of the following values:

  1. The character is an alpha character.

  2. The character is a digit.

  3. The character is an accented character.

  4. The character is equivalent to a space.

  5. The character is a control character, and will be replaced by a space during indexing.

Any characters whose codes are not present in the utf8class.txt file are given a nextrieve classification value of zero -- ie, they will be treated as punctuation.

foldfilename

A foldfilename specifies a file whose content defines the case-folding of unicode data. This information is used to map words containing characters differing in case to the same word in the index.

Each line of the file should start with the unicode code point (in hex) of an ``alpha'' type character followed by a space, then followed by a space-separated sequence of unicode code points (in hex) of the character's lower-case equivalent. Normally there is just one entry in this sequence. Accents should be preserved with this transformation.

decompfilename

A decompfilename specifies a file whose content defines the decomposition of an accented character to a non-accented form.

Each line of the file should start with the unicode code point (in hex) of an accented (or decomposable) character, followed by a space and a space-separated sequence of unicode code points (in hex) of the decomposition. Accents should be removed in this transformation.

Note that it is possible to decompose the unicode code point for 1/4, for example, into the sequence ``1'', ``/'', and ``4''.
